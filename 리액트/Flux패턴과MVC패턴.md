## ✨Flux패턴에 대해서 설명해주세요.

Flux는 기존의 MVC 모델의 단점을 보완하기 위해 페이스북에서 발표한 디자인 패턴인데,

`사용자 입력을 기반으로 Action을 만들고 Action을 Dispatcher에 전달하여 Store(Model)의 데이터를 변경한 뒤 View에 반영하는 단방향의 흐름인 디자인 패턴`입니다.

MVC 모델에서는 View가 다양한 상호작용을 위해 여러 개의 Model을 동시에 업데이트하고 Model 역시 여러 개의 View를 업데이트할 수 있는데, 새로운 기능이 추가되거나 규모가 커질수록 복잡한 데이터 흐름을 가지게 되어 예측 불가능한 코드를 만들게 되고 버그가 발생하게 되었다.

Flux 패턴의 등장은 이런 맥락에서 예측 가능성을 높이기 위해 나왔다고 볼 수 있습니다. 각 요소들이 단방향 흐름에 따라 순서대로 역할을 수행하고, View에서 새로운 변경이 생기면 다시 같은 순서로 실행함으로서 흐름을 예측할 수 있게 된다는 장점이 있습니다.

## 🤔Flux 패턴의 장단점을 아시는대로 말씀해주세요.

장점으로는

- 어플리케이션의 규모가 커지면 데이터의 연속적인 갱신이 되는 상황에서는 사용자 상호작용의 결과가 어떤 변화를 만드는지 예측하는데 어려워집니다.

- 갱신으로 인한 데이터 변경이 단 한 차례만 이뤄진다면 전체 시스템은 좀 더 예측 가능하게 됩니다.

- 데이터 흐름이 한 방향으로 강제되기 때문에, 양방향 흐름으로 인한 혼선을 줄일 수 있고, 모든 상태가 스토어에 있음으로 변경 사항을 여러 컴포넌트에 전달하기도 쉽습니다.

단점이라고 한다면

1. 모든 상태 변경을 엄격하게 관리하기 때문에 작은 규모에서도 과도한 상태 관리가 생길 수 있고,

2. 단방향이기 때문에 코드의 복잡성이 다소 증가할 수 있습니다.

3. 그리고 초기에는 Action, Dispatcher, Store 등의 개념을 이해하기 위한 약간의 러닝커브 또한 있을 수 있습니다.

## 🤔 Flux패턴의 구성요소에 대해서 설명해주세요 (ex 디스패처가 뭔가요?)

1. 액션(Action) : 어떠한 작업을 처리할 액션과 그 액션 발생 시 함께 포함시킬 데이터를 의미합니다. 액션 타입과 데이터를 각 정의해 이를 디스패처로 보냅니다.

2. 디스패처(dispatcher) : 액션을 스토어에 보내는 역할을 합니다. 콜백 함수 형태로 앞서 액션이 정의한 타입과 데이터를 모두 스토어에 보냅니다.

3. 스토어(store) : 여기에서 실제 상태에 따른 값과 상태를 변경할 수 있는 메서드를 가지고 있습니다. 액션의 타입에 따라 어떻게 변경할지가 정의되어 있습니다.

4. 뷰 : 리액트에서는 컴포넌트에 해당하는 부분입니다. 스토어에서 만들어진 데이터를 가져와 화면을 렌더링하는 역할을 합니다.

또한 뷰에서도 사용자의 입력이나 행위에 따라 상태를 업데이트 하고자 할 때는 뷰에서 액션을 호출하는 구조로 구성됩니다.

요약하면, action은 disaptcher를 통해 store의 상태를 갱신합니다. store가 업데이트 된 후, 상태가 변경되었다는 이벤트를 중계하는 과정으로 view에게 새로운 상태를 보내주고 view 스스로 업데이트하게 만듭니다.

## 🤔 Redux와 Flux는 무슨 관계가 있나요?

redux에 대해 정의를 간단히 해보자면 flux패턴을 이용하여 상태 관리를 할 수 있도록 도와주는 라이브러리라고 할 수 있습니다.

flux 패턴과 다른 점은 reducer의 존재인데,

1. view에서 dispatch를 이용하여 action을 reducer에 전달

2. reducer에서는 action에 맞는 행동을 취하며 store에 데이터를 저장

3. store에 저장되어 있는 데이터를 view에서 가져다 사용하게 됩니다.

## 🤔Redux가 어떻게 구성되고 실행되는지 알려주세요.

1. 먼저, 상태(state)를 변경하기 위한 액션(Action) 객체를 생성합니다.

-이 액션 객체는 type 속성과 필요한 추가 데이터를 포함합니다.

2. 이후, 생성된 액션 객체를 스토어(Store)로 디스패치(dispatch)하면, 스토어는 현재 상태와 액션을 리듀서(Reducer)로 전달합니다.

3. 리듀서는 현재 상태와 액션을 받아 새로운 상태를 반환하는 순수 함수입니다.

4. 마지막으로, 리듀서가 반환한 새로운 상태는 스토어에 저장되고, 스토어는 구독된 모든 리스너(listener)에게 상태 변경을 알립니다.

5. 이를 통해 리액트 컴포넌트는 상태 변경을 감지하고 필요에 따라 다시 렌더링됩니다.

이러한 과정을 통해 리덕스는 애플리케이션 상태를 예측 가능하고 일관되게 관리할 수 있습니다.

## 🤔MVC 패턴은 무엇인가요?

데이터를 관리하는 Model, 사용자 인터페이스를 담당하는 View, 그리고 사용자 입력을 처리하는 Controller로 구성된 디자인 패턴입니다.

MVC 패턴은 각각의 구성요소가 독립적으로 작동하여 유지보수와 확장성을 향상하고
애플리케이션의 논리적인 구조를 분리함으로써 유연성을 높이고 개발 속도를 빠르게 할 수 있다.

MVC 모델에서는 View가 다양한 상호작용을 위해 여러 개의 Model을 동시에 업데이트하고 Model 역시 여러 개의 View를 업데이트할 수 있는데, 새로운 기능이 추가되거나 규모가 커질수록 복잡한 데이터 흐름을 가지게 되어 예측 불가능한 코드를 만들게 되고 버그가 발생하게 되었다.
